# План рефакторинга YOJO Generator

**Цель**: Улучшить поддерживаемость, тестируемость и расширяемость кодовой базы, сохраняя совместимость генерации с `src/test/resources/example/expected`.

**Дата**: 2026-05-04

---

## 1. Анализ текущего состояния

### 1.1 Основные проблемы
1. **Смешение ответственностей**: В одном классе (например, `AbstractMapper`) находятся логика парсинга YAML, маппинга типов, обработки аннотаций и генерации кода.
2. **Статические утилиты**: Огромные классы `MapperUtil` и `LombokUtils` с статическими методами, которые трудно тестировать и которые создают сильную связность.
3. **Монолитная генерация кода**: Методы `Schema.toWrite()` и `Message.toWrite()` (по ~600 строк) содержат всю логику формирования Java-кода, включая импорты, аннотации, JavaDoc.
4. **Сложная условная логика**: В `AbstractMapper.fillVariableProperties()` огромная цепочка `if-else`, обрабатывающая разные типы свойств (массивы, карты, ссылки, полиморфизм и т.д.).
5. **Обработка ошибок**: Использование `System.err.println` и бросание `RuntimeException`/`IllegalArgumentException` вперемешку.
6. **Работа с файлами**: Логика записи файлов разбросана в `YojoGenerator`.
7. **Полиморфизм и дискриминаторы**: Логика разбросана между `SchemaMapper`, `AbstractMapper` и самими доменными объектами.
8. **Контекст и состояние**: `ProcessContext` передается в каждый метод, содержит много полей, некоторые из которых мутируются.

### 1.2 Ключевые атрибуты (Dictionary)
- **YAML ключи**: `properties`, `type`, `format`, `enum`, `$ref`, `allOf`, `oneOf`, `anyOf`, `additionalProperties`, `discriminator`, `const`, `x-enumNames`, `x-class-annotation`, `x-field-annotation` и многие другие.
- **Java типы и форматы**: Маппинг из `format` (например, `date` → `LocalDate`, `uuid` → `UUID`, `big-decimal` → `BigDecimal`).
- **Аннотации**: Jakarta/javax validation (`@NotNull`, `@NotBlank`, `@Size`, `@Digits`, `@Pattern`, `@Email`, `@Valid`), Jackson (`@JsonTypeInfo`, `@JsonSubTypes`, `@JsonTypeId`), Lombok (`@Data`, `@AllArgsConstructor` и т.д.).
- **Коллекции**: `realization` (ArrayList, HashSet и т.д.), `collectionType` (list, set).
- **Наследование**: `extends`, `implements`, `fromClass`, `fromPackage`, `fromInterface`.
- **Прочее**: `validationGroups`, `validationGroupsImports`, `validateByGroups`, `pathForGenerateMessage`, `removeSchema`, `default`, `example`, `description`, `title`, `summary`, `multipleOf`, `digits`, `minLength`/`maxLength`, `minimum`/`maximum`, `pattern`.

Все эти атрибуты должны корректно обрабатываться после рефакторинга.

## 2. Этапы рефакторинга

### Этап 1: Выделение генерации кода (Code Generation Separation)
**Паттерн**: *Strategy* / Просто выделение ответственности.

**Зачем**: Отделить логику формирования Java-исходника от доменных объектов.

**Действия**:
1. Создать пакет `ru.yojo.codegen.generator.code`.
2. Создать классы:
   - `SchemaCodeGenerator` (принимает `Schema`, возвращает `String`).
   - `MessageCodeGenerator` (принимает `Message`, возвращает `String`).
   - Возможно, абстрактный `AbstractCodeGenerator` с общими методами (генерация импортов, JavaDoc, геттеров/сеттеров).
3. Перенести методы `Schema.toWrite()` и `Message.toWrite()` в соответствующие генераторы.
4. В доменных объектах оставить только данные (DTO). Можно добавить метод `generateCode(CodeGenerator)`, но лучше, чтобы сервис-генератор вызывался извне.
5. **Преимущества**: Легче менять шаблоны генерации, проще тестировать генерацию отдельно от маппинга.

### Этап 2: Применение паттерна Strategy для обработки типов свойств
**Паттерн**: *Strategy* + *Chain of Responsibility*.

**Зачем**: Заменить цепочки `if-else` в `AbstractMapper.fillVariableProperties()` на набор взаимозаменяемых обработчиков.

**Действия**:
1. Определить интерфейс `PropertyTypeHandler`:
   ```java
   public interface PropertyTypeHandler {
       boolean canHandle(Map<String, Object> propertyMap, VariableProperties vp);
       void handle(String schemaName, VariableProperties vp, 
                  Map<String, Object> currentSchema, Map<String, Object> schemas,
                  String propertyName, Map<String, Object> propertiesMap,
                  ProcessContext processContext, Map<String, Object> innerSchemas);
   }
   ```
2. Создать конкретные обработчики (каждый в своем файле):
   - `ObjectTypeHandler` (для объектов с `properties`)
   - `ArrayTypeHandler` (для `type: array`)
   - `MapTypeHandler` (для `additionalProperties`)
   - `ReferenceTypeHandler` (для `$ref`)
   - `EnumTypeHandler` (для перечислений)
   - `PolymorphicTypeHandler` (для `oneOf`/`allOf`/`anyOf`)
   - `ExistingObjectHandler` (для `format: existing`)
3. В `AbstractMapper` (или новом `PropertyTypeResolver`) создать список обработчиков и метод `resolveType()`, который проходит по цепочке, находит подходящий обработчик и вызывает его.
4. **Преимущества**: Легко добавлять новые типы, каждый обработчик тестируется отдельно, устраняется сложная вложенность условий.

### Этап 3: Выделение парсинга спецификации
**Паттерн**: *Facade* или просто выделение сервиса.

**Зачем**: Инкапсулировать чтение YAML и сбор схем/сообщений.

**Действия**:
1. Создать класс `AsyncAPIParser` (или `SpecificationAnalyzer`), который:
   - Принимает путь к файлу спецификации.
   - Читает YAML (через SnakeYAML).
   - Собирает `schemas` и `messages` из корневого документа и внешних `$ref`.
   - Возвращает объект `SpecificationModel` (или просто пару `Map<String, Object>`).
2. Перенести методы `collectSchemasAndMessages`, `collectExternalRefs` из `YojoGenerator` в этот класс.
3. **Преимущества**: `YojoGenerator` станет проще, логика парсинга централизована.

### Этап 4: Внедрение Builder-ов для доменных объектов
**Паттерн**: *Builder*.

**Зачем**: Упростить и сделать надежным создание сложных объектов `Schema`, `Message`, `VariableProperties`.

**Действия**:
1. Для `Schema` создать `SchemaBuilder` с методами:
   ```java
   Schema schema = Schema.builder()
       .name("User")
       .packageName("com.example.common;")
       .addField(VariableProperties...)
       .lombokProperties(lombokProps)
       .build();
   ```
2. Аналогично для `Message` и `VariableProperties`.
3. Использовать билдеры в мапперах.
4. **Преимущества**: Читаемость, гарантия обязательных полей, избежание состояния "объект создается частями".

### Этап 5: Улучшение обработки полиморфизма (дискриминаторы)
**Паттерн**: *Strategy* или *Visitor*.

**Зачем**: Вынести логику обработки дискриминаторов в отдельный компонент.

**Действия**:
1. Создать класс `DiscriminatorProcessor`, который инкапсулирует:
   - Предварительное сканирование базовых схем.
   - Поиск подтипов.
   - Добавление аннотаций `@JsonTypeInfo`/`@JsonSubTypes`.
2. Использовать его в `SchemaMapper`.
3. **Преимущества**: Сложная логика полиморфизма изолирована, легче поддерживать и расширять.

### Этап 6: Централизованная обработка ошибок
**Зачем**: Заменить `System.err.println` и разрозненные исключения на систему ошибок.

**Действия**:
1. Создать иерархию исключений в `ru.yojo.codegen.exception` (например, `SpecificationParseException`, `CodeGenerationException`, `SchemaValidationException`).
2. Вместо `System.err.println` использовать логирование (SLF4J) или хотя бы централизованный `ProblemReporter`.
3. **Преимущества**: Единообразие, возможность точно обрабатывать ошибки.

### Этап 7: Рефакторинг утилит (`MapperUtil`, `LombokUtils`)
**Зачем**: Уменьшить связность и улучшить тестируемость.

**Действия**:
1. Часть методов из `MapperUtil` (например, работа с именами `capitalize`, `toValidJavaFieldName`, `refReplace`) вынести в отдельный `NamingStrategy` или `NameConverter`.
2. Логику Lombok из `LombokUtils` инкапсулировать внутри самого `LombokProperties` или выделить в `LombokAnnotationBuilder`.
3. Остальные утилиты (касты, проверки) оставить статическими, но сгруппировать по назначению и, возможно, сделать методами экземпляра, если они используют состояние (например, `ProcessContext`).
4. **Преимущества**: Меньше статических зависимостей, проще тестировать.

### Этап 8: Работа с файловой системой
**Зачем**: Изолировать логику записи файлов.

**Действия**:
1. Создать `JavaFileWriter` (или `SourceCodeWriter`), который умеет:
   - Создавать директории.
   - Записывать файлы с правильной кодировкой (UTF-8).
   - Возможно, поддерживать "dry run" или возврат сгенерированного кода без записи (для тестирования).
2. Использовать его в `YojoGenerator` или в новом оркестраторе.
3. **Преимущества**: Легче тестировать генерацию без файловой системы.

### Этап 9: Обработка `innerSchemas` (внутренних схем)
**Зачем**: Инкапсулировать логику регистрации внутренних схем.

**Действия**:
1. Создать `InnerSchemaRegistry`, который будет:
   - Хранить карту внутренних схем.
   - Определять, нужно ли генерировать схему или это примитив.
   - Обеспечивать уникальность имен.
2. Использовать его в мапперах.
3. **Преимущества**: Логика регистрации централизована, уменьшается риск дублирования.

### Этап 10: Уменьшение статики и DI (Dependencies)
**Зачем**: Сделать код более тестируемым, уменьшить связность.

**Паттерн**: *Dependency Injection* (простейшая форма — передача зависимостей через конструктор).

**Действия**:
1. Сделать `AbstractMapper` (и его наследников) stateful: хранить `schemasMap`, `innerSchemas`, `processContext` как поля экземпляра, а не передавать в каждый метод.
2. Конвертировать статические утилиты из `MapperUtil` в методы экземпляра (или оставить статическими, но группировать по назначению).
3. Вместо создания `SchemaMapper` и `MessageMapper` в `YojoGenerator` "вручную", можно использовать простую фабрику или просто конструкторы с параметрами.
4. **Преимущества**: Легче писать юнит-тесты с моками, ясные зависимости.

## 3. Порядок выполнения рефакторинга

Рекомендую выполнять этапы последовательно, после каждого этапа запуская тесты (чтобы генерация оставалась совместимой с `src/test/resources/example/expected`).

1. **Этап 1** (выделение генерации кода) — наименее рискованный, сразу улучшает структуру.
2. **Этап 8** (файловый writer) — можно сделать параллельно с Этапом 1 или сразу после него.
3. **Этап 3** (парсинг) — желательно сделать до большого рефакторинга мапперов.
4. **Этап 4** (билдеры) — можно внедрять по мере изменения мапперов.
5. **Этап 2** (стратегии для типов) — можно делать постепенно, переводя по одному типу за раз.
6. **Этап 5** (дискриминаторы) — можно выделить в любой момент, так как логика уже частично отделена.
7. **Этап 9** (inner schemas) — после Этапа 2 и 3, когда мапперы стабилизируются.
8. **Этап 6** (ошибки) — можно делать параллельно с другими этапами.
9. **Этап 7** (утилиты) — после основных структурных изменений.
10. **Этап 10** (DI) — проводить параллельно с другими этапами, когда меняются конструкторы.

## 4. Как проверять совместимость

После каждого значительного изменения запускать:
```bash
./gradlew test
```
Тесты сравнивают сгенерированный код с ожидаемым в `expected`. Допускаются незначительные отклонения (порядок импортов, пробелы), но суть должна остаться.

**Важно**: Если тесты падают из-за незначительных расхождений (например, перестановка импортов), можно обновить ожидаемые файлы, если это не меняет семантику.

## 5. Дополнительные рекомендации

- Использовать **Java 17+ records** для небольших неизменяемых объектов (например, для пары "имя-значение" в перечислениях).
- Рассмотреть возможность использования **sealed interfaces** для иерархии обработчиков (если используется Java 17+).
- Для генерации кода рассмотреть **шаблонизатор** (например, Freemarker, как указано в roadmap), но это большой шаг, лучше сначала стабилизировать структуру.
- Добавить **Snapshot Testing** инфраструктуру: если после рефакторинга код изменился (например, переставились импорты), тесты должны показать diff.
